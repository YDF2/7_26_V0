# 2026-03-23 视觉检测逻辑与类别 ID 一致性检查

## 结论摘要

在你给定的数据集约束下：

1. ball 的 id=1，post 的 id=0。
2. 在线运行链路（controller + Vision + TRTDetector）当前类别映射是正确的。
3. 存在的主要逻辑风险不在在线检测主链，而在离线工具 auto_marker 的默认映射仍是反的（0=ball,1=post）。

因此：

1. 在线无框问题，不是由 Vision 中 ball/post id 写反直接导致。
2. 但你用 auto_marker 验证 engine 时，可能被错误映射误导结论。

## 在线检测链路一致性

## 1) Vision 侧类别映射

Vision 构造中：

1. ball_id_=1
2. post_id_=0

该设置与“数据集 ball=1、post=0”一致。

## 2) TRTDetector 侧类别读取

在 TRTDetector::detect 中，类别概率读取是：

1. prob_post = field(4 + post_id_)
2. prob_ball = field(4 + ball_id_)

当 post_id_=0、ball_id_=1 时，分别读取第 4/5 个类别分量，逻辑正确。

## 3) 结果显示链路

Vision 在 IMAGE_SEND_RESULT 分支中：

1. ball_dets_ 非空则画蓝框。
2. post_dets_ 非空则画红框。
3. 最终 send_image 发送 JPG 到 debuger。

所以只要检测结果非空，Qt 就能看到框。

---

## 发现的风险点（与“无框”强相关）

## 风险 A：auto_marker 默认映射与在线不一致（高）

auto_marker 目前默认：

1. detector.load(engine_path, 0, 1)
2. 注释写明 class 0=ball, class 1=post

这与当前在线 Vision 的 1/0 映射相反。

影响：

1. 你离线验证 engine 时可能得到“看起来没问题/有问题”的错误结论。
2. 进而误判在线无框原因。

## 风险 B：输出布局被硬编码为 2 类（中）

TRTDetector::parseOutputLayout 中固定 expected_fields=6（4 + 2 类）。

如果你实际 engine 不是 2 类输出：

1. detect 会返回 false。
2. Vision 会清空 det，表现为无框。

## 风险 C：阈值与过滤较严格（中）

在线检测还叠加以下过滤：

1. ball/post 置信度阈值。
2. 最小尺寸阈值。
3. 球宽高比约束 d_w_h。
4. NMS。

即使 id 正确，也可能被过滤成 0 个目标。

---

## 判断结果

“ball=1，post=0”这一前提下，在线检测主逻辑没有发现 id 映射错误。

但存在一个会严重干扰排查判断的逻辑不一致：

1. auto_marker 默认映射与在线 Vision 不一致。

建议至少保证离线验证时与在线一致（ball=1、post=0），再评估 engine 与阈值问题。

---

## 建议的最小动作

1. 离线验证时统一类别映射为 ball=1、post=0。
2. 增加在线每 N 帧日志：ball_dets/post_dets 数量与 top prob。
3. 若仍长期 0，优先检查 engine 输出类别数与当前 parseOutputLayout 假设是否一致。
